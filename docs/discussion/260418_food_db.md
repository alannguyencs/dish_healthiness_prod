## 1. Curated Nutrition Database Lookup

### Data

- **Unified CSV:** `data/nutrition_db/unified_nutrition_db.csv` — 4,493 records, 8 columns: `original_db_name, original_db_id, dish_name, calories, carb, protein, fat, per_unit`.
- **Source DBs (all shipped as CSV under `resources/db/`):**
  - `Anuvaad_INDB_2024.csv` — Indian, 1,014 records.
  - `ciqual_2020.csv` — French, 3,186 records (per-100g).
  - `malaysian_food_calories.csv` — 60 records.
  - `myfcd_basic.csv` + `myfcd_nutrients.csv` — MyFCD (Malaysian) split into basic info and long-format nutrients (233 records, re-nested at load time).
- **Coverage:** calories 78%, carb 94.6%, protein 96.0%, fat 96.1%.
- **Doc:** `docs/nutrition_db_unified.md`.

### Code

- Service class: `src/service/collect_from_nutrition_db.py::NutritionCollectionService`.
- On init it:
  1. Validates each CSV exists (`DatabaseNotFoundError` otherwise).
  2. Loads each DB as `List[Dict]` (`_load_malaysian_database`, `_load_myfcd_database`, `_load_anuvaad_database`, `_load_ciqual_database`). MyFCD is reconstructed from the two CSVs with a per-`ndb_id` join so each record carries a nested `nutrients` dict keyed by nutrient name with `value_per_100g` / `value_per_serving` / `unit` / `category`.
  3. Builds a **BM25 index per database** (`_build_bm25_search_indices`) over a synthesized searchable document per record (`_create_searchable_document`) that includes the food name, category, English name (for CIQUAL), and hand-crafted synonym/variation expansions (`_generate_food_variations`, `_extract_clean_terms_from_myfcd`, `_extract_clean_terms_from_anuvaad`, `_generate_indian_food_variations`).
- Public entry point: `collect_from_nutrition_db(text, min_confidence=70, deduplicate=True)` — convenience function instantiating the service and calling `_search_dishes_direct`.

**Key point:** it is a **real structured lookup**, not CV-only estimation. Every image-analysis flow that claims "DB-backed" actually calls this service; the DB results are then fed into the consolidation prompt.

## 2. Multi-Stage Database Search

Two levels of "multi-stage" exist — one inside the retrieval method and one at the orchestration layer.

### 2a. Retrieval-method stages (inside `NutritionCollectionService`)

Documented in `docs/nutrition_db_retrieval_methods.md`. Four methods were implemented and benchmarked on the 846-query eval set (`data/nutrition_db/retrieval_eval_dataset.csv`) with NDCG@10, MRR, Recall@10:

| Method | Library | Overall NDCG@10 (measured/expected) |
|---|---|---|
| BM25 (lexical) | `rank-bm25` | **0.7744 — winner** |
| Fuzzy (edit-distance) | `rapidfuzz` | rejected |
| OpenAI embeddings | `text-embedding-3-large`, vectors saved to `data/nutrition_db/embedding/` | rejected |
| Hybrid (BM25 35% + Fuzzy 25% + Embeddings 40%, RRF) | — | designed, not selected |

Live code uses **BM25** (`_search_dishes_direct` → `_direct_bm25_search`). The loser methods are kept as scripts/docs for the benchmark.

### 2b. Confidence scoring inside `_direct_bm25_search`

Per candidate the confidence combines a keyword-match component and a BM25-quality component:

- **Keyword match** is an 85/15 weighted mix of "core dish-token match ratio" and "descriptor-token match ratio", with additive bonuses of +0.20 when 2+ core tokens match and a further +0.15 when 3+ match (perfect-match boost).
- **BM25 quality** is the candidate's raw BM25 score logarithmically normalized against the maximum raw score in the result batch — this spreads scores and prevents one outlier from collapsing the distribution.
- **Final confidence** is `0.8 × keyword_match + 0.2 × BM25_quality`, then linearly scaled into the `[0.50, 0.95]` range so no match returns a deceptively confident 1.0 or a useless 0.0.

Quantity descriptors (`1…9`, `small/large/medium/big/mini`, `wraps/pieces/servings`) are stripped from the "core" dish tokens so they don't inflate ratios. The calling side stores `_current_dish_tokens` so dish-name tokens can be weighted higher than the generic descriptor tokens of `related_keywords`.

Each returned row includes: `matched_food_name, source, confidence (0–1), confidence_score (0–100), nutrition_data, raw_bm25_score, matched_keywords, total_keywords`. Results are then deduplicated (highest-confidence wins; BM25 score as tiebreaker) and ranked.

### 2c. Orchestration stages (inside `AIAgentService._extract_and_lookup_nutrition`)

`src/service/ai_agent.py` wraps the retrieval with a fallback strategy:

- **Stage 1 — individual dish names.** For each dish name emitted by OpenAI/Gemini image analysis, call `collect_from_nutrition_db(dish_name, min_confidence=70)`. Track `best_confidence`, best result, and a `search_attempts` log.
- **Stage 2 — combined terms.** If `best_confidence < 0.75`, re-query with `", ".join(all_food_items)` at `min_confidence=60` and replace the best result only if the combined search scores higher.
- **Selection.** The winning result is annotated with `search_strategy ∈ {"individual_dish_name: <name>", "combined_terms: <joined>"}` plus `search_attempts` and `dish_candidates` for auditability.

The consolidation prompt (`resources/ai_consolidation_prompt.md`, section "Nutrition Database Search Strategy") also tells the LLM exactly that this two-stage, confidence-ranked search was performed, and treats `confidence ≥ 80%` as "high-confidence" when deciding whether to let the DB override the LLM's per-nutrient estimate.

There is also a richer entry `NutritionCollectionService.search_nutrition_database_enhanced(dish_name, related_keywords, estimated_quantity, top_k)` used by the "DB-access-before-analysis" flows (Flow 4/5 in `docs/analysis_flow.md`), which tokenizes the dish name as priority tokens, adds descriptor tokens from the comma-separated keywords, removes duplicates, and performs OR-logic retrieval across all four DBs.

## 3. Multi-Source Evidence Fusion

Fusion is implemented as a single LLM call with structured output, not as a numeric ensemble. Three sources, fixed priority:

| Priority | Source | Produced by |
|---|---|---|
| 1 | **Nutrition DB matches** (with confidence scores) | `NutritionCollectionService` → `_extract_and_lookup_nutrition` |
| 2 | **LLM image analysis** (dish name, keywords, calories/macros/fiber/micronutrients, healthiness score+rationale) | `openai_analyzer.analyze_with_openai_async` or `gemini_analyzer.analyze_with_gemini_async` |
| 3 | **Personalization** from similar prior user dishes (optional) | similarity match or user manual input (`_get_personalized_analysis`) |

### Orchestration

`src/service/ai_agent.py::AIAgentService.generate_ai_agent_analysis_split_async`:

1. Step 0: personalized analysis (reuses `existing_personalized` when available; otherwise skipped — there is no "AI-generated fake personalization").
2. Step 1/2: OpenAI and Gemini image analyses (parallel, each reusable from cache).
3. Steps 3+4: **OpenAI path and Gemini path in parallel**, each:
   - calls `_extract_and_lookup_nutrition` on its own analysis only (so the two agents don't contaminate each other),
   - calls `AIConsolidationService.consolidate_analyses_openai` or `consolidate_analyses` with all three priority sources plus `user_id`, model name, and the consolidation prompt.
4. Returns `{openai_agent, gemini_agent, nutrition_data_openai, nutrition_data_gemini, processing_info(step_times, phase_times)}`.

Timing summary printed per run separates "Initial Analysis" from the parallel "Consolidation Phase" and reports time saved by parallelization.

### Consolidation service

`src/service/ai_consolidation.py::AIConsolidationService`:

- **Structured output:** Pydantic `ConsolidatedAnalysis` with flattened `reasoning_*` fields (Gemini's structured-output mode can't handle nested schemas). `to_dict()` re-nests reasoning into a JSON string for the frontend.
- **Gemini path:** `client.models.generate_content(response_mime_type="application/json", response_schema=ConsolidatedAnalysis, temperature=0.1)` → `response.parsed`, with text-parse fallback.
- **OpenAI path:** `client.responses.parse(text_format=ConsolidatedAnalysis, reasoning={"effort": low|medium|high})` with `gpt-5` (reasoning-mode variants).
- **Prompt assembly (`_build_consolidation_prompt`):** base prompt + JSON-dumped OpenAI result + JSON-dumped Gemini result + JSON-dumped nutrition DB result + personalization (only if `_has_meaningful_personalized_data(...)` **and** `_source == 'user_manual_input'`; otherwise an explicit "NO USER PERSONALIZED DATA PROVIDED" sentinel is inserted so the LLM doesn't hallucinate).
- **Alternative entry `consolidate_with_extraction_reasoning`:** template-substitution prompt (`__nutrition_database__`, `__llm_initial_results__`, `__personalized__`, `__llm_provider__`) with the image attached to the consolidation call — used by Flow 6/7 (DB-access-after-analysis).

### Fusion rules (from `resources/ai_consolidation_prompt.md`)

Per-nutrient decision logic the LLM is told to follow:

- If a **high-confidence DB match (≥80%)** exists → use it to refine the LLM number; use the image to validate + portion-scale; personalization only as tertiary.
- If some DB data exists but none ≥80% → weigh DB vs LLM using the image; personalization tertiary.
- If no DB data → use LLM image analysis; personalization only to refine.
- If only personalization → use it but state limitations.

Integrity checks:

- **Atwater cross-check** `calories ≈ 4·carbs + 4·protein + 9·fat` with ±15% tolerance; otherwise adjust the least certain macro and record the adjustment in `reasoning_*`.
- Non-negative integer nutrients.
- Each `reasoning_*` field must attribute the selected value to its source and explain conflicts/scaling.

## Implications for this project (`dish_healthiness_prod`)

The earlier audit (`docs/discussion/260418_key_technical_features_status.md`) concluded all three items are **Not Done** here. The good news is that the ground truth from `/Volumes/wd/projects/dish_healthiness` is already **partially present** in this repo:

- `backend/resources/database/` already ships the same source CSVs (`Anuvaad_INDB_2024.csv`, `ciqual_2020.csv`, `malaysian_food_calories.csv`, `myfcd_basic.csv`, `myfcd_nutrients.csv`) plus `conversion_stats.json` / `ciqual_2020_stats.json`.
- The current analysis pipeline is Gemini-only, two-step (Step 1 component ID → Step 2 nutrition), orchestrated by `backend/src/api/item_step1_tasks.py` and `backend/src/api/item_tasks.py`, using `analyze_step2_nutritional_analysis_async` with prompts from `backend/resources/step2_nutritional_analysis.md`. There is no retrieval layer and no fusion layer.

Below is a concrete per-feature adaptation plan that reuses the reference project's code and artifacts rather than redesigning from scratch.

### Phase 3 — Consolidation (detailed design)

"Phase 3" is a direct extension of the existing **Phase 1 (Component Identification)** and **Phase 2 (Nutritional Analysis)** background-task pattern already in this repo (`backend/src/api/item_step1_tasks.py`, `backend/src/api/item_tasks.py`, shared error plumbing in `backend/src/api/_phase_errors.py`). It's the job that fuses DB evidence + Step 2 output + optional personalization into the final consolidated estimate described in Feature 3.

**Why it's a separate phase, not inline in Phase 2**

- Phase 2 already hits `gemini-2.5-pro` with `thinking_budget=-1`; adding another LLM call (with the image re-attached) inside that task would double its wall-clock time and make retry semantics awkward.
- The nutrition DB lookup is deterministic and cheap (~ms); the consolidation is the expensive LLM step. Splitting them lets us retry consolidation alone without re-paying Phase 2, mirrors the existing `step1_error` / `step2_error` error blocks, and keeps each phase under the "one responsibility, one retry unit" rule the current pipeline already follows.
- It also opens the door to re-consolidate when the user edits personalization (manual corrections) — the DB matches and Step 2 output don't change, only the third priority source does.

**Trigger.** From inside `trigger_step2_analysis_background` (in `item_tasks.py`), once `update_dish_image_query_results(...)` for `step2_data` has committed, fire Phase 3 as a non-awaited background task — the HTTP caller should see a fast "step2 done" completion and the consolidated result lands in a subsequent poll/refresh. This matches how Phase 1 hands off to Phase 2 today. The Step 1.5 DB lookup itself stays synchronous and fast inside the user-confirmation endpoint; Phase 3 is specifically the LLM-based fusion call.

**Background task — file layout.** New file `backend/src/api/item_consolidation_tasks.py` parallel to `item_step1_tasks.py` / `item_tasks.py`, exposing `trigger_consolidation_background(query_id, image_path, retry_count=0)`. It reads `result_gemini.nutrition_db_matches` (written by Step 1.5), `result_gemini.step2_data` (written by Phase 2), and any user manual corrections, calls the consolidation service with the image attached, writes `step3_data` back into `result_gemini`, and clears any prior `step3_error`. On exception it delegates to `persist_phase_error(query_id, exc, retry_count, "step3_error")` just like the other two phases do.

**Persistence shape — reuse `result_gemini`, no new column.** The current pipeline persists both Step 1 and Step 2 output as keys inside the existing `result_gemini JSONB` column (see `item_step1_tasks.py` and `item_tasks.py`). Phase 3 follows the same convention so no schema migration is required: a top-level `step` counter advances 0 → 1 → 2 → 3; new top-level keys `nutrition_db_matches` (from Step 1.5) and `step3_data` (consolidated nutrients + reasoning JSON string) are added alongside the existing `step1_data` / `step2_data` / `step1_confirmed` / `iterations` / `current_iteration`. Only if a column-level split is wanted later (indexing, partial updates), promote `step3_data` to a dedicated `result_consolidated JSONB`.

**Iterations mirroring.** The existing Phase 1/2 code writes `step2_data` into `result_gemini["iterations"][current_iteration-1]` so the history view in `dish_query_iterations.py` stays complete. Phase 3 must do the same for `step3_data` — otherwise the iteration history UI will show a gap.

**Error handling — reuse the shared phase-error plumbing.** In `backend/src/api/_phase_errors.py`, `persist_phase_error(query_id, exc, retry_count, "step3_error")` is a zero-code-change reuse (the helper is already parameterized by `error_key`). The existing classifier (`classify_phase_error`) already handles `config_error`, `image_missing`, `parse_error`, `api_error`, `unknown` — those buckets apply verbatim to Phase 3 since it's the same Gemini client and same structured-output parse path. The only additions are: a new user-facing fallback message for `step3_error`, and extending `PhaseErrorCard` on the frontend to read `step3_error` alongside `step1_error` / `step2_error`. **Critical fallback property:** a Phase 3 failure must not invalidate Step 2 — the Item view still renders Step 2 numbers; the consolidated tab shows a "Consolidation unavailable — retry" banner. Ensured by (a) writing `step3_error` without touching `step2_data`, and (b) the frontend reading `step3_data` only when `step === 3 && step3_error == null`.

**Retry.** The existing `backend/src/api/item_retry.py` exposes Step-1 / Step-2 retries. Add a `retry_step3` endpoint that simply re-invokes `trigger_consolidation_background(query_id, image_path, retry_count+1)`. Because Phase 3 reads `nutrition_db_matches` + `step2_data` from the persisted record, no upstream re-computation is needed — this is the operational payoff of the split.

**Inputs fed to the consolidation prompt.** Map each existing field to the prompt placeholder (`backend/resources/ai_consolidation_prompt.md` after copy):

| Prompt placeholder | Source field | Priority |
|---|---|---|
| `__nutrition_database__` | `result_gemini.nutrition_db_matches` (JSON-dumped) | 1st |
| `__llm_initial_results__` | `result_gemini.step2_data` (JSON-dumped) | 2nd |
| `__personalized__` | User manual corrections (if any), else `**NO USER PERSONALIZED DATA PROVIDED**` | 3rd |
| `__llm_provider__` | Literal `"Gemini"` (this repo has no OpenAI path) | n/a |

The **image must be attached** to the Gemini call (pattern: reference project's `_consolidate_with_gemini_extraction` uses the genai `Part.from_bytes` for image input alongside the prompt text). Without the image, the Atwater check + portion-scaling instructions in the prompt degrade to guesswork.

**Timing budget.** Reference project measurements (`docs/analysis_flow.md` timing summary):

| Phase | Typical latency | Notes |
|---|---|---|
| Nutrition DB lookup | < 50 ms | Synchronous, 4 BM25 indices in memory |
| Consolidation (Gemini 2.5 Flash) | 2–5 s | Image + ~8 KB prompt |
| Consolidation (Gemini 2.5 Pro) | 5–10 s | Matches current Phase 2 model |

Given Phase 2 already uses `gemini-2.5-pro`, consolidation can use `gemini-2.5-flash` to cut ~5 s off user-visible "analysis complete" latency — the prompt is tight and structured-output is strongly constrained, so Flash is usually sufficient. Leave the model name configurable (`GEMINI_CONSOLIDATION_MODEL` env var), defaulting to Flash.

**Frontend surface.** `frontend/src/pages/ItemV2.jsx` already renders Step 2 results. Add a new "AI Agent" / "Consolidated" tab that appears only when `step === 3`, showing the consolidated `calories_kcal / fiber_g / carbs_g / protein_g / fat_g / micronutrients`; an expandable "Reasoning" section rendering each `reasoning_*` key (Sources / Calories / Fiber / Carbs / Protein / Fat / Micronutrients); and a "Top DB matches" chip row reading `nutrition_db_matches.nutrition_matches[0..4]` with `confidence_score` badges so users can see the evidence the LLM was given.

**API exposure.** `backend/src/api/item.py` currently returns `result_gemini` straight through. No change is strictly required — the new `step3_data` / `nutrition_db_matches` / `step3_error` keys ride along. If the frontend prefers a flatter contract, add a light shaping helper that surfaces `{ step1, step2, step3, nutrition_db_matches, step1_error, step2_error, step3_error }` at the top level of the response.

**Acceptance criteria for Phase 3 being "Done"**

1. A fresh image upload reaches `result_gemini.step === 3` with a valid `step3_data` payload that passes the Atwater ±15% check.
2. `nutrition_db_matches` is populated for that query and is referenced in `step3_data.reasoning`.
3. Killing the Phase 3 task mid-flight leaves `step2_data` intact, surfaces a `step3_error` block, and the "Retry consolidation" button re-runs Phase 3 alone (no re-call of Step 1 or Step 2).
4. Updating a user manual correction on the item re-triggers only Phase 3 and produces a new `step3_data` without touching the earlier steps.

Key invariants the diagram encodes:

1. **Clean phase split.** Phase 1 is strictly about *what is this and how much of it?* (identification + portion count). Phase 2 is strictly about *given the confirmed dish and portion, what are the nutrients?* — the user never re-confirms anything between 2.1/2.2/2.3.
2. **Reference is a hint, not ground truth.** Phase 1.1.1's retrieved reference image + its prior `step1_data` are passed to Phase 1.1.2 as a second image + an explicit "Reference results" block. The prompt frames them as hints because the two dishes may differ; the LLM is expected to disagree when the query image doesn't match. Cold-start users simply have `reference_image = null` and Phase 1.1.2 runs single-image.
3. **Write-after-read for the personalized index.** Phase 1.1.1 reads the per-user BM25 index *before* it inserts the current image+description, so the current upload never matches itself. Inserting at the end also guarantees the next upload by the same user benefits from this one.
4. **Phase 2.1 and Phase 2.2 run in parallel** and both complete before Phase 2.3 fires. They write to independent JSONB keys (`nutrition_db_matches`, `personalized_matches`) so neither blocks the other, and either can be empty without breaking the pipeline.
5. **Threshold-gated integration.** Phase 2.3 is the single LLM call that produces the final numbers. DB matches and personalization are *references*, not *overrides* — each is only injected into the prompt if its top-confidence / top-similarity crosses a configurable threshold. Below threshold, the block is omitted so low-quality retrievals can't drag the estimate around.
6. **Phase independence.** Each phase reads from and writes to `result_gemini JSONB` and can be retried alone. A Phase 2.3 failure leaves `nutrition_db_matches` and `personalized_matches` intact, so a retry only re-runs the Gemini call.
7. **User corrections replay Phase 2.3 only.** A manual edit on the Item view re-triggers Phase 2.3 (fast path) with the existing 2.1/2.2 artifacts; Phase 1 and the retrievals are not re-run.

### Suggested phasing

1. **Phase A (retrieval only, no user-visible change):** port `nutrition_db.py` + benchmark test. Unblocks feature 1 and feature 2 as library code.
2. **Phase B (wire into pipeline):** add `nutrition_db_matches` persistence + call it between Step 1 confirmation and Step 2.
3. **Phase C (fusion):** port `ai_consolidation.py` + prompt, add Phase-3 background task, surface `result_consolidated` to the frontend.
4. **Phase D (optional):** backfill the benchmark (846-query eval set) as a regression gate in CI, and add a second LLM (OpenAI) path for A/B parity with the reference architecture.

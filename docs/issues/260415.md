# Implementation plan — End-to-end workflow from `docs/discussion/260418_food_db.md`

**Target diagram:** the ASCII "End-to-end workflow diagram" section (lines ~295–500) of `docs/discussion/260418_food_db.md`.

**Scope:** split the full Phase 1 + Phase 2 redesign into independently shippable stages so each can be reviewed and tested in isolation. Stages are ordered by dependency; within a stage, items are ordered for a single-PR implementation.

Legend: **[NEW]** = create new module/table/column. **[UPD]** = modify existing module/prompt/endpoint. **[CFG]** = wiring/config only. Each stage ends with an explicit "done when" acceptance line so it can be closed without ambiguity.

- [x] **Stage 2 — Phase 1.1.1: fast caption + personalized reference retrieval**

    Depends on Stage 0. First user-visible change is invisible (only a new `reference_image` key appears in `result_gemini`) — wiring for stages 3 and 7.

    - **[NEW]** `backend/src/service/llm/fast_caption.py` — `generate_fast_caption_async(image_path) -> str`. Model `gemini-2.0-flash`, temperature 0, plain-text response (no structured schema). Reuses `os.environ["GEMINI_API_KEY"]` and the existing `loop.run_in_executor` pattern from `gemini_analyzer.py`.
    - **[NEW]** `backend/src/service/personalized_reference.py` — orchestrator `resolve_reference_for_upload(user_id, query_id, image_path) -> {description, reference_image | None}`:
        - call `generate_fast_caption_async`
        - `tokens = personalized_food_index.tokenize(description)`
        - `matches = personalized_food_index.search_for_user(user_id, tokens, top_k=1, min_similarity=THRESHOLD_PHASE_1_1_1)` (exclude self by `query_id`)
        - join top-1 row back to its `DishImageQuery.result_gemini` to pull `prior_step1_data`
        - **insert AFTER search** into `personalized_food_descriptions` so the current upload never matches itself
        - return `{description, reference_image: {query_id, image_url, description, similarity_score, prior_step1_data} | None}`
    - **[UPD]** `backend/src/api/item_step1_tasks.py::analyze_image_background` — before the Step 1 Gemini call, call `resolve_reference_for_upload` and stash result on `result_gemini.reference_image` (new top-level key). Persist even if `reference_image = null` so the frontend can tell cold-start from unpersisted.
    - **[CFG]** `backend/src/configs.py` — `THRESHOLD_PHASE_1_1_1_SIMILARITY = 0.25` (BM25 normalized; tune after first runs).
    - **Done when:** uploading two photos under the same user causes the second upload's `result_gemini.reference_image` to point at the first; the first upload's `reference_image` is `null`. Schema migration applied to dev DB.

## Cross-stage invariants the implementation must preserve

1. **Phase independence on `result_gemini`.** A Phase 2.3 failure must not invalidate `nutrition_db_matches` or `personalized_matches` — retry only re-runs the Gemini call.
2. **Reference is a hint, not ground truth.** Phase 1.1.2 and Phase 2.3 prompts explicitly frame both reference images and reference blocks as hints.
3. **Write-after-read for the per-user index.** Phase 1.1.1 searches the index *before* inserting the current row, so the upload never matches itself.
4. **Threshold-gated inclusion.** DB matches, personalization blocks, and reference-image-B are each omitted below their respective threshold — never passed through as low-quality hints.
5. **No breaking change to the existing `/api/item/{id}` contract.** All new data rides on additional `result_gemini` keys (`reference_image`, `nutrition_db_matches`, `personalized_matches`, `step2_corrected`); frontend reads them defensively.
6. **Cold-start must work.** Every stage must degrade cleanly to today's behavior when the user has no history and no DB match crosses threshold.

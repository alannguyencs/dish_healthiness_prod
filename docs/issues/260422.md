# Rename plan — remove `step1` / `step2` terminology

**Goal:** eliminate `step1` / `step2` / `Step1` / `Step2` from the codebase. The two concepts have domain names already — adopt them consistently:

- **Step 1 → "identification"** (dish & component identification + serving-size predictions, Phase 1.1.2)
- **Step 2 → "nutrition"** (nutritional analysis: calories, macros, micros, healthiness, Phase 2.3)

**Scope survey (at plan time):**

|---|---|
| `backend/src/service/llm/` | 35 |

| `backend/tests/` | 390 |
| `frontend/` | 295 |
| `docs/` | 1,248 |
| other backend | 17 |
| **Total** | **2,088** |

Legend: **[REN]** = rename symbol/file, no behavior change. **[MIG]** = schema or persisted-data change, requires a write/read compat step. **[UPD]** = call-site follow-through.

## Cross-tier invariants

1. **Tier 1 must land before Tier 2.** JSONB keys reference symbol names only loosely, but dragging both renames through a single PR makes review intractable.
2. **Tier 2 is a deploy-coupled PR.** Backend + frontend + DB migration ship together; no feature-flag gate.
3. **Zero silent data loss.** The migration must be idempotent and reversible. Stage it against a copy of prod before the real run.
4. **Tier 3 never blocks a release.** Docs can lag code by one merge.

## Suggested PR shape

- **PR-1 (Tier 1):** symbols + filenames + prompt `.md` renames. All tests green. No DB, no frontend.
- **PR-2 (Tier 2):** JSONB migration + backend write-path rename + frontend lockstep rename. Coordinated deploy.
- **PR-3 (Tier 3):** docs sweep. Merge anytime after PR-2.

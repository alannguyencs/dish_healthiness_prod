# MealSnap Feature Checklist

## Multimodal AI for Healthiness and nutrition assessment

- [x] **Photo-Based Dish Analysis**: The AI agent analyzes an uploaded **dish image** directly, using multimodal vision to assess what food is present.
- [x] **Automatic Dish Identification**: It determines the most likely **dish name** by combining image evidence with structured nutrition database matches.
- [x] **Healthiness Category**: It produces a consolidated **healthiness category label** based on the final nutritional interpretation.
- [x] **Healthiness rationale**: An explanation why the dish got that label is also provided.
- [x] **Calorie estimation**: Total calories for the meal in kilocalories.
- [x] **Fiber estimation**: Total dietary fibre in grams.
- [x] **Carbohydrate estimation**: Total carbohydrates in grams.
- [x] **Protein estimation**: Total protein in grams.
- [x] **Fat estimation**: Total fat in grams.
- [x] **Notable micronutrients**: A list of noteworthy vitamins and minerals the meal contains.

## Refining the results with manual corrections for difficult cases

- [x] **Set the dish name**: The user can pick one of our application's proposed names or type a completely custom name when the suggestions are wrong.
- [x] **Include or exclude detected components**: Each automatically detected components has a checkbox for inclusion.
- [x] **Change the portion size**: The portion size for each component can be swapped via a dropdown.
- [x] **Adjust portions**: The user can type a custom portion description when none of the suggestions fit and change how many portions of each component they actually ate.
- [x] **Add a missed component**: Components our application failed to detect can be added manually.
- [x] **Serving Size Helpful reference Guide**: A standalone reference page with common portion sizes to help the user to add better manual correct.

## Dish Recognition and Quantity Estimation

- [x] **Dish name recognition**: Our application automatically analyses an uploaded photo and proposes up to five possible names for the overall dish, each with a likelihood score.
- [x] **Individual-dish breakdown**: Our application lists the individual components visible on the dish, up to ten.
- [x] **Portion size suggestions**: For the detected components, our application proposes three to five realistic portion size estimations (suggestions).
- [x] **Predicted number of portions**: For every detected component, our application estimates how many portions are in the photo.

## Dishes history calendar and Logging dishes

- [x] **Monthly calendar view**: A month-at-a-glance grid of everything the user has logged.
- [x] **Up to five dishes per day**: Every calendar day offers five fixed slots for logging separate dishes.
- [x] **Photo upload**: A dish can be logged by picking an image file from the device or by pasting the web address of a photo.

## Key In-Depth Technical Features

- [ ] **Curated Nutrition Database Lookup**: The agent queries a **structured nutrition database of dishes and nutrient values**, rather than relying only on computer vision direct estimation.
- [ ] **Multi-Stage Database Search**: It uses composed search strategies in multiple stages to better find relevant information for nutritional evaluation as well as confidence scores to adequately rank query results.
- [ ] **Multi-Source Evidence Fusion**: It combines **database results, AI image analysis, and optional prior user-like dish information** into one unified estimate.
- [ ] **Cooking Method and Regional Variant Context**: In the analysis information about the **cooking style, preparation method, and regional variant** are taken into account.
- [x] **Nutritional Consistency Evaluation**: Technical internal guidelines for adequate nutritional assessment are applied to ensure higher plausibility in the estimated outputs.

# Known Issues on Shipped Features

Issues discovered while reviewing the checked items above. Grouped by severity.

## Should fix (functional / user-visible)

- [x] **Phase 2 has no error UI**: If the Gemini Step 2 call fails, the frontend sits on "Calculating nutritional values…" forever — no retry, no error toast, no surfaced state. Affects all 8 Phase-2 items.
- [x] **Phase 1 has no error UI** (follow-up to the Phase 2 fix): apply the same `step1_error` + `POST /retry-step1` pattern that landed for Phase 2 in `docs/plan/260418_phase2_error_handling.md`. Shipped in `docs/plan/260418_phase1_error_handling.md` — Phase 1 + Phase 2 now share `_phase_errors.py` (backend) and `PhaseErrorCard.jsx` (frontend); the Phase 1 background task moved to `backend/src/api/item_step1_tasks.py`; polling logic extracted to `frontend/src/hooks/useItemPolling.js`.
- [x] **No debounce or idempotency on `POST /confirm-step1`**: Double-tapping Confirm enqueues two Phase-2 background tasks that race; last-writer-wins. The button disables via `isConfirming` but the server has no guard.
  - Fixed: new `confirm_step1_atomic` CRUD acquires a row-level lock (`SELECT … FOR UPDATE`) and only flips `step1_confirmed` once. The endpoint schedules the Phase-2 background task only on the winner; the loser gets HTTP 409 (`ItemV2` treats 409 as benign — resumes polling, no error toast). Tested in `backend/tests/test_item_confirm.py`.
- [x] **Re-uploading a slot creates orphan rows**: No dedupe at DB level. The GET endpoint returns one row, the duplicates leak. Affects "Up to five dishes per day".
  - Fixed: new `replace_slot_atomic` CRUD locks any existing row(s) for `(user_id, date(target_date), dish_position)`, deletes them, and inserts the new row in one transaction. Both `/api/date/.../upload` and `/upload-url` now go through it and best-effort delete the orphaned image files from disk. A partial UNIQUE index `uq_dish_image_query_prod_dev_user_target_position` was added in `backend/sql/create_tables.sql` as a DB-level guard against concurrent upload races. Tested in `backend/tests/test_date_upload.py`.
- [ ] **Wording mismatch on "Automatic Dish Identification"**: PDF says "combining image evidence with structured nutrition database matches" — there is no DB. Functionally checked, literally inaccurate. Either build the DB lookup or reword the PDF line.

## Worth tightening (silent footguns)

- [x] **Pricing fallback is silent**: If the Gemini model is swapped to one not in `pricing.py`'s `PRICING` table, `compute_price_usd` falls back to `DEFAULT_PRICING` ($0.075 / $0.30 per M tokens) — cost numbers are quietly wrong. Affects every checked item that relies on cost reporting.
  - Fixed: `compute_price_usd` now logs a `WARNING` the first time each unknown `(vendor, key)` falls back to `DEFAULT_PRICING` (de-duped via a module-level set so we don't spam every Gemini call). The fallback still happens — pricing should never crash a billing path — but ops can grep `"No pricing entry"` to spot a mis-named model.
- [x] **`GEMINI_API_KEY` missing → frontend polls forever**: Background task raises and is swallowed; no surfaced error to the user. Affects Phase 1 and Phase 2.
  - Already fixed by the Phase 1 + Phase 2 error-UI work (`docs/plan/260418_phase1_error_handling.md`, `260418_phase2_error_handling.md`). `gemini_analyzer` raises `ValueError("GEMINI_API_KEY environment variable is required")`, the background task catches it, `_phase_errors.classify_phase_error` maps it to `"config_error"`, and `PhaseErrorCard` renders the configuration error with retry disabled. `useItemPolling` stops on `step1_error`/`step2_error` so the spinner does not loop forever.
- [x] **Filename collision on same-second uploads**: Image filename pattern is `{yyMMdd_HHmmss}_dish{N}.jpg`. Two uploads into the same slot in the same second would clobber. Unlikely but unguarded.
  - Fixed: filenames now include the user id — `{yymmdd_HHMMSS}_u{user_id}_dish{N}.jpg`, built by a single `_build_image_filename` helper used by both `/upload` and `/upload-url`. Cross-user collisions in the same second are impossible; same-user same-slot same-second collisions are now resolved by `replace_slot_atomic` (which deletes the prior row + image first).
- [x] **`MAX_DISHES_PER_DATE = 5` is API-only, not a DB constraint**: A non-browser client could insert more than five rows for a given day.
  - Fixed: added an idempotent `CHECK (dish_position IS NULL OR dish_position BETWEEN 1 AND 5)` constraint (`ck_dish_image_query_prod_dev_dish_position_range`) in `backend/sql/create_tables.sql`. Combined with the new `(user_id, date(target_date), dish_position)` UNIQUE index, the DB layer caps rows at five per `(user, day)`.
- [x] **`serving_sizes` schema floor is `min_length=1`** while the prompt asks Gemini for 3-5. The schema permits a degenerate single-option response. Tighten to `min_length=3` if the 3-5 guarantee matters server-side.
  - Fixed: `ComponentServingPrediction.serving_sizes` is now `min_length=3, max_length=5`. A Gemini response with fewer than three options now fails Pydantic validation, gets classified as `parse_error`, and surfaces via the existing Phase 1 error UI instead of slipping through as a degenerate single option.

## Cosmetic / by-design

- [x] **Healthiness category thresholds are hardcoded in `Step2Results.jsx`** (≥81 / ≥61 / ≥41 / ≥21). Changing them is a frontend edit, not an API change.
  - Fixed: extracted `HEALTHINESS_TIERS` (cutoff + label + color) and `resolveHealthinessTier(score)` into `frontend/src/utils/healthiness.js`. `Step2Results.jsx` now reads from there, so changing a threshold is a one-line edit in one file instead of two parallel `if`-ladders.
- [x] **Macronutrients are integers** — no decimals on calories, fiber, carbs, protein, fat.
  - Fixed: `Step2NutritionalAnalysis` now types `calories_kcal/fiber_g/carbs_g/protein_g/fat_g` as `float` (still `ge=0`). Gemini can return decimals; frontend display is unchanged (renders the value as-is). Existing integer-valued rows keep validating because Python `int` is a subtype of `float` for Pydantic.
- [x] **Micronutrients are free-form strings**, no canonical list and no mg quantities.
  - Fixed: introduced a `Micronutrient` Pydantic model `{name: str, amount_mg: Optional[float]}` and changed Step 2 `micronutrients` to `List[Micronutrient]`. New rows carry an optional mg quantity (rendered as `"Vitamin C (45mg)"`); `Step2Results.jsx` keeps a backward-compat branch so legacy rows stored as `List[str]` still render. A canonical vocabulary is still deliberately not enforced — the LLM picks names freely — but the shape is now structured.
- [x] **`number_of_servings` floor mismatch**: frontend clamps to `0.1`, backend Pydantic accepts down to `0.01`. No real impact, but inconsistent.
  - Fixed: backend Pydantic floors raised from `ge=0.01` to `ge=0.1` in both `ComponentConfirmation` (`item_schemas.py`) and `ComponentServingPrediction` (`models.py`), matching the frontend `min="0.1"` clamp.

- [x] **Wire tests into pre-commit and/or CI**: shipped as two new local hooks in `.pre-commit-config.yaml` — `pytest-backend` (gated on `^backend/.*\.py$`) and `jest-frontend` (gated on `^frontend/src/.*\.(js|jsx)$`). Both use `pass_filenames: false` so they run the full suite once per relevant change instead of per-file. Trade-off accepted: commits get slower but every commit is gated on green tests.
